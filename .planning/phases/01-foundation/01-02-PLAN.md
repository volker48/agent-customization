    Create `pi-extensions/pi-rlm/index.ts` as the default export function receiving `ExtensionAPI`.

    **Helper: `formatResult<T>(result: RlmResult<T>): string`** — format the `{ ok, data, meta }` envelope as a readable text string for the LLM content field. Apply `truncateHead` to the formatted output. Set `meta.truncated` flag if truncation occurred.

    **Helper: `makeDetails(store: DocumentStore, toolName: string): object`** — per research recommendation, only `rlm_load` and `rlm_save` emit full snapshots (they mutate store state). `rlm_search`, `rlm_extract`, `rlm_get` emit minimal details (no snapshot). This keeps session file size bounded.

    **Tool: `rlm_load`**
    - Parameters: `source` (string, required — file path or raw content), `type` (optional StringEnum of DocType), `name` (optional string), `chunk_size` (optional integer), `overlap` (optional integer), `batch` (optional array of `{ source, type?, name? }`)
    - When `batch` is provided, use `addDocumentBatch`. When only `source` is provided, use `addDocument`.
    - File path detection: if `source` starts with `/` or `~` or `./`, read from filesystem via `node:fs/promises`. Otherwise treat as raw content.
    - When loading from file path and `type` not provided, infer from extension: `.py` → python, `.ts`/`.tsx` → typescript, `.js`/`.jsx` → javascript, `.md` → markdown, else text. Per user decision.
    - Return `{ ok: true, data: { doc_id, chunk_count, total_chars }, meta: { truncated: false } }` formatted as text content.
    - Include full store snapshot in `details` (mutating operation).
    - Error handling: file not found → `{ ok: false, data: null, meta: { error: "File not found: /path" } }`. Batch failure → similar with which item failed.

    **Tool: `rlm_search`**
    - Parameters: `query` (string, required), `limit` (optional integer, default 10)
    - Call `store.search(query, { limit })`.
    - Format results as text listing: each result shows chunk_id, doc_name, score, and a text excerpt.
    - Return `{ ok: true, data: results, meta: { truncated, result_count, query } }`.
    - Minimal details (no snapshot — read-only operation).

    **Tool: `rlm_extract`**
    - Parameters: `chunk_id` (optional string), `doc_id` (optional string — required with char_start/char_end), `char_start` (optional integer), `char_end` (optional integer)
    - If `chunk_id` provided: call `store.extractByChunkId(chunk_id)`. Return chunk text.
    - If `doc_id` + `char_start` + `char_end` provided: call `store.extractByRange(doc_id, char_start, char_end)`. Return extracted text.
    - If neither valid combination: return error.
    - Return `{ ok: true, data: { text, chunk_id?, doc_id?, char_start?, char_end? }, meta: { truncated } }`.
    - Minimal details (read-only).

    **Tool: `rlm_save`**
    - Parameters: `name` (string, required), `value` (any, required — JSON-serializable per Claude's Discretion)
    - Call `store.saveBuffer(name, value)`.
    - Return `{ ok: true, data: { name, saved: true }, meta: { truncated: false } }`.
    - Include full store snapshot in `details` (mutating operation).

    **Tool: `rlm_get`**
    - Parameters: `name` (string, required)
    - Call `store.getBuffer(name)`.
    - If found: return `{ ok: true, data: { name, value }, meta: { truncated } }`.
    - If not found: return `{ ok: false, data: null, meta: { error: "Buffer not found: name" } }`.
    - Minimal details (read-only).

    **Truncation:** Apply `truncateHead` (from `@mariozechner/pi-coding-agent`) to every tool's text content output using `DEFAULT_MAX_LINES` and `DEFAULT_MAX_BYTES`. Append truncation notice if truncated, matching the pattern in `exa-search.ts`.

<task type="auto">
  <name>Task 2: Wire lifecycle events for branch-safe state reconstruction</name>
  <files>pi-extensions/pi-rlm/index.ts</files>
  <action>
    In the same `index.ts` file, add state reconstruction logic after tool registration.

    **`reconstructState(ctx: ExtensionContext)`:**
    - Call `store.reset()` to clear all in-memory state.
    - Walk `ctx.sessionManager.getBranch()` entries.
    - For each entry: check `entry.type === "message"` and `entry.message.role === "toolResult"`.
    - Only process `rlm_load` and `rlm_save` tool results (the two tools that emit snapshots).
    - Extract `details.snapshot` from the tool result message. If present, call `store.restore(snapshot)`.
    - After walking all entries, the store reflects the last mutating operation on this branch.

    **Register all four lifecycle events** (per research — missing `session_tree` is a known pitfall):
    ```
    pi.on("session_start", async (_event, ctx) => reconstructState(ctx));
    pi.on("session_switch", async (_event, ctx) => reconstructState(ctx));
    pi.on("session_fork", async (_event, ctx) => reconstructState(ctx));
    pi.on("session_tree", async (_event, ctx) => reconstructState(ctx));
    ```

    **Key implementation details:**
    - `getBranch()` returns only entries on the current branch path. When forked, the new branch's history ends at the fork point — reconstruction automatically yields the pre-fork snapshot. No special fork handling needed (per research Pattern 1).
    - Process entries in order (oldest first). The last valid snapshot wins — this handles the case where multiple `rlm_load` calls occurred.
    - Silent rehydration per user decision: no notification to the agent that state was restored.

    **ExtensionContext type:** Import from `@mariozechner/pi-coding-agent`. The `ctx` parameter in lifecycle callbacks provides `ctx.sessionManager.getBranch()`.
  </action>
